# Experiment 11: Adversarial Context Injection for TriviaQA

Tests whether adversarial context contamination (wrong answers from the same category injected into the prompt window) degrades LLM QA performance on TriviaQA. Uses a three-condition design with length-matched context windows.

## Research Question

Does the presence of adversarially injected wrong answers in the context window degrade LLM performance on TriviaQA factoid QA tasks, and is any degradation attributable to adversarial contamination specifically (vs. total context length)?

## Conditions

| Condition | Description |
|-----------|-------------|
| Scoped | Only the current question. No history. Length-padded to match Contaminated. |
| Full-History | Complete prior-session transcript verbatim prepended to the question. Length-padded to match Contaminated. |
| Contaminated | Question injected at a controlled middle position among distractor turns containing wrong answers from the same category. No padding needed (reference length). |

## Key Methodological Controls

### Length-Matching Approximation

Scoped and Full-History conditions are padded with filler text to match the token count of the Contaminated condition. The filler text is lorem-style prose with no numeric patterns. Token counts use a whitespace-split approximation, which is not equivalent to tiktoken or Claude's internal subword tokenizer.

### Wrong Answer Generation

Wrong answers for the contaminated condition are generated by category-matching against `answer.type`. When the type field is empty or None, a random answer from the dataset is selected. Gold answers are always excluded from the candidate pool.

### Answer Matching

Predicted answers are matched against all `normalized_aliases` (not just `normalized_value`) using case-insensitive comparison. This accounts for legitimate answer variations (e.g., "NYC" vs "New York City").

### Task-ID Keyed Alignment

Paired records in scoring are aligned by `task_id` key, not positional zip. This ensures that missing tasks in one condition do not cause misalignment in McNemar tests.

## Usage

```bash
# Dry run (print context lengths without API calls)
uv run python run_inference.py --dry-run

# Pilot run (30 tasks)
uv run python run_inference.py --pilot --condition scoped --model gemma4
uv run python run_inference.py --pilot --condition contaminated --model gemma4

# Full run (300 tasks, one condition)
uv run python run_inference.py --condition scoped --model gemma4
uv run python run_inference.py --condition full-history --model gemma4
uv run python run_inference.py --condition contaminated --model gemma4

# Score
uv run python score.py
```

## Gate Criteria

The pilot gate (issue #49 pre-registered stopping rule) is evaluated after the contaminated pilot run:

- Compute scoped accuracy and contaminated accuracy on the pilot sample (n=30 per condition)
- If gap (scoped - contaminated) < 8 percentage points: print STOP (contamination effect insufficient)
- If gap >= 8 percentage points: print PROCEED (contamination effect confirmed, proceed to full scale)

Full-scale gate criteria:
- McNemar test p < 0.025 (Bonferroni-corrected) for scoped vs contaminated
- Error rate <= 0.05 per condition
